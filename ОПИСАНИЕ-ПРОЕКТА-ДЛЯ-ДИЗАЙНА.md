# Описание проекта «Мотт Ларбе» для дизайна

Документ предназначен для создания дизайна продукта в отдельном проекте (Claude и др.). Содержит точное описание текущего состояния backend, целевой функциональности и карты страниц.

---

## 1. Суть продукта

**Название:** Мотт Ларбе (языковая платформа).

**Цель:** Платформа для изучения языка через чтение реальных текстов с мгновенным переводом по клику, морфологическим разбором и персональным словарём.

**Ключевая идея:** Чтение текста = интерактивное изучение. Каждое слово — точка входа в словарь: клик → перевод, базовая форма, грамматическая форма. Без переключения вкладок и поиска в словаре.

**Решаемые проблемы:**
- Постоянное переключение между текстом и словарём
- Потеря контекста при переводе
- Сложность морфологии (особенно редкие/малоизвестные языки)
- Неудобство работы с адаптированными текстами

## 3. Архитектура backend (модули)

Backend построен как набор движков:

| Движок / модуль | Назначение |
|------------------|------------|
| **Text** | Тексты, страницы, версии, хранение Tiptap JSON и plain-текста |
| **Tokenizer** | Разбиение текста на токены (слова), привязка к страницам |
| **Normalizer** | Приведение слов к нормализованной форме (lowercase, без пунктуации) |
| **Dictionary (Admin)** | Админский словарь: Lemma, Headword, MorphForm, Sense, DictionaryEntry |
| **Dictionary Cache** | Кэш переводов по нормализованной форме для быстрого поиска |
| **Online Dictionary** | Интеграция внешнего словаря (импорт/поиск) |
| **Morphology** | Морфологический анализ, базовая форма, правила (Rule Engine, Importer, Cleaner) |
| **Unknown Word** | Логирование слов, которых нет в словаре (для расширения словаря) |
| **Token** | Информация по токену: перевод, грамматика, базовая форма (в т.ч. кэш) |
| **Words** | Lookup по `tokenId` (клик по слову) и lookup по строке (`lookup-by-word`) |
| **Progress** | Прогресс по тексту и по словам (UserTextProgress, UserWordProgress) |
| **Auth** | Login, Register, Refresh, Logout, JWT, роли USER/ADMIN |
| **User** | Профиль, обновление, удаление аккаунта |

Обработка текста после сохранения: извлечение plain-текста из Tiptap → токенизация → нормализация → поиск в словаре/морфологии/кэше → связь токенов со словарём (TextVocabulary, TokenAnalysis).

## 4. Модель данных (основные сущности)

### Пользователи и роли
- **User:** id, email, password, username, name, surname, phone?, role (USER | ADMIN), refresh token
- **UserRole:** USER, ADMIN

### Тексты
- **Text:** id, title, level?, language (CHE | RU), author, source?, createdById
- **TextPage:** id, textId, pageNumber, contentRich (Tiptap JSON), contentRaw (plain)
- **TextProcessingVersion:** версия обработки текста (textId, version)

### Токены и словарь текста
- **TextToken:** id, versionId, pageId?, position, original, normalized, status (ANALYZED | AMBIGUOUS | NOT_FOUND), vocabId? → связь с TextVocabulary
- **TextVocabulary:** на уровне версии текста: versionId, normalized, lemmaId?, translation? (связь токенов с леммами и переводами)

### Словарь (админ + импорт)
- **Lemma:** базовая форма слова (baseForm, normalized, language, partOfSpeech?, frequency?)
- **DictionaryEntry:** запись из внешнего/админского словаря (rawWord, rawTranslate, notes, source: ADMIN | IMPORT | ONLINE | CACHE)
- **Headword:** форма входа в словарь (entryId, text, normalized, lemmaId?)
- **MorphForm:** словоформа (form, normalized, grammarTag?, lemmaId, entryId?)
- **Sense:** значение/определение (entryId, order, definition, notes)
- **Example:** пример к значению (senseId, text, translation?)

### Кэш и анализ
- **DictionaryCache:** быстрый поиск по normalized (normalized, lemmaId?, translation?)
- **TokenAnalysis:** какой леммой разобран токен (tokenId, lemmaId?, source: ADMIN | CACHE | ONLINE | MORPHOLOGY, probability?, isPrimary?)

### Неизвестные слова
- **UnknownWord:** word, normalized, seenCount, firstSeen, lastSeen (уникальность по normalized)

### Прогресс пользователя
- **UserTextProgress:** userId, textId, progressPercent, lastOpened
- **UserWordProgress:** userId, lemmaId, status (NEW | LEARNING | KNOWN), seenCount, repetitions, lastSeen, nextReview, easeFactor?, interval? (для интервального повторения)

## 6. API (краткая сводка для дизайна)

Базовый префикс: **`/api`**. Защищённые маршруты: **`Authorization: Bearer <access_token>`**. Refresh — в cookie.

| Группа | Метод | Путь | Назначение |
|--------|-------|------|------------|
| Auth | POST | `/api/auth/login` | Вход |
| Auth | POST | `/api/auth/register` | Регистрация |
| Auth | POST | `/api/auth/login/access-token` | Обновление access по refresh |
| Auth | POST | `/api/auth/logout` | Выход |
| Users | GET | `/api/users/:id` | Профиль по ID |
| Users | PATCH | `/api/users` | Обновить свой профиль |
| Users | DELETE | `/api/users` | Удалить аккаунт |
| Texts | GET | `/api/texts` | Список текстов |
| Texts | GET | `/api/texts/:id` | Текст со всеми страницами |
| Texts | GET | `/api/texts/:id/pages/:pageNumber` | Одна страница + токены + прогресс |
| Texts | POST | `/api/texts` | Создать текст (Admin) |
| Texts | PATCH | `/api/texts/:id` | Обновить текст (Admin) |
| Texts | DELETE | `/api/texts/:id` | Удалить текст (Admin) |
| Words | POST | `/api/words/lookup` | Перевод по tokenId (клик по слову) |
| Words | POST | `/api/words/lookup-by-word` | Перевод по строке (normalized) |
| Tokens | GET | `/api/tokens/:id` | Информация о токене |
| Progress | GET | `/api/progress/text/:id` | Прогресс по тексту для пользователя |
| Admin | POST | `/api/admin/dictionary` | Создать запись в словаре (Admin) |

*Планируются эндпоинты:* админ — токены, Unknown Words, морфология, пользователи, подписки; приложение — личный словарь, повторения, статистика по словам.

Детальные контракты (тела запросов/ответов) — в файле **`api-requests.md`** в корне репозитория.

## 7. Ключевые сценарии для UI/UX

### 7.1 Чтение текста и клик по слову
1. Пользователь открывает текст (Library → Text details → Reader).
2. Видит страницу с интерактивными словами (контент + токены с `/api/texts/:id/pages/:pageNumber`).
3. Кликает по слову → frontend вызывает **POST `/api/words/lookup`** с `{ "tokenId": "..." }`.
4. Backend: поиск по tokenId → TextToken → TextVocabulary / Lemma / морфология / кэш; при отсутствии в словаре — запись в UnknownWord (seenCount+1).
5. Ответ: `translation`, `grammar`, `baseForm` — показываются во всплывающем окне (Word popup) или в боковой панели (Word panel).

### 7.2 Неизвестное слово
- Если слово не найдено: в ответе пустые/нулевые поля; на backend создаётся/обновляется UnknownWord.
- В админке раздел **Unknown words**: список (слово, нормализованная форма, количество появлений, контекст/текст, даты). Действия: добавить в словарь, связать с существующей леммой, удалить.

### 7.3 Админ: тексты
- Список текстов: название, уровень, язык, количество слов, дата, статус публикации.
- Создание/редактирование: title, language, level, author, source, страницы (Tiptap JSON).
- Отдельно: управление версиями текста, при необходимости — переразметка (re-tokenize).

### 7.4 Админ: словарь и морфология
- Словарь: поиск, карточка слова, добавление/редактирование перевода, грамматики, словоформ.
- Морфологические правила: список, добавление/редактирование, тест анализа.

### 7.5 (Планируется) Редактирование отдельного слова в тексте
- В админ-редакторе текста: клик по слову → карточка токена (original, normalized, position, vocabId). Можно изменить слово, нормализованную форму, привязку к словарю без полной переразметки текста.

## 8. Обязательные улучшения (из бэклога)

Эти пункты нужно учитывать в дизайне и приоритизации:

1. **Полная структура админ-панели:** тексты (создание, версии, публикация, уровень A1–C2), словарь, Unknown words, морфологические правила, пользователи, подписки/платежи.
2. **Механизм клика по слову:** описан выше; важно отображение перевода/грамматики/базовой формы и логирование Unknown Word.
3. **Подписки:** бесплатный тариф (ограниченный перевод/личный словарь), премиум (без лимитов, статистика, повторения, сложные тексты). Варианты: Basic, Premium, Pro.
4. **Личный словарь пользователя:** слово, перевод, дата добавления, уровень изучения, количество повторений; просмотр, удаление, отметка «выучено», статистика.
5. **Система повторения (интервальное):** уровни знания, дата следующего повторения, интервалы (1 → 3 → 7 → 14 → 30 дней); формат карточек (слово ↔ перевод).
6. **Редактирование отдельных токенов в тексте:** без полной переразметки; массовое исправление при необходимости; опция «Re-tokenize text» только при сильном изменении текста.

## 13. Instructions to tailor Claude's responses

When working on design for this project, tailor your responses as follows:

**Context & scope**
- You are designing for **Мотт Ларбе** — a language-learning platform (reading texts, click-to-translate, morphology, personal vocabulary). Primary languages: **CHE** (Chechen), **RU** (Russian). Levels: **A1–C2**.
- Assume the backend is NestJS + Prisma + PostgreSQL; frontend is Next.js + Tiptap. API is REST under `/api`; auth is JWT (access token + refresh in cookie).
- When proposing UI or flows, align with the **page map** (Public, Auth, App, Admin) and the **API** described in this document and in `api-requests.md`. Do not invent endpoints or pages that contradict them unless explicitly asked to suggest new ones.

**Terminology**
- Use consistent terms: **текст** / **text**, **токен** / **token** (word in text), **лемма** / **lemma** (base form), **словарь** / **dictionary**, **неизвестные слова** / **Unknown words**, **прогресс** / **progress**, **Reader** (reading view), **Word popup** / **Word panel**.
- Admin areas: **Texts**, **Dictionaries**, **Morphology**, **Unknown words**, **Users**, **Payments** — use these names in navigation and labels.

**Output format**
- Prefer **Russian** for user-facing copy and design descriptions unless the user asks for another language.
- For UI specs: describe layout, components, and states (empty, loading, error, success); mention key actions and where they lead.
- For flows: use short step-by-step or bullet lists; reference concrete screens (e.g. Library → Text details → Reader → Word popup).
- If listing API calls, use method and path (e.g. `POST /api/words/lookup`) and note auth (Bearer / Admin) when relevant.

**Constraints**
- Respect **obligatory improvements** (admin structure, click-on-word flow, subscriptions, personal dictionary, spaced repetition, token editing) as part of the product vision; treat them as required when suggesting features or priorities.
- Acknowledge technical limits: complex morphology, need for fast lookup (cache), and scalability of dictionary and long texts.

**When uncertain**
- Prefer asking one short clarifying question rather than inventing details that might conflict with the backend or the page map.
- If the user attaches or references files (e.g. `api-requests.md`, `Полная карта всех страниц.md`), use them as the source of truth for API and navigation.
